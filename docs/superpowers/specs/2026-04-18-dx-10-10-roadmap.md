# Greentic Designer Extensions — DX 10/10 Roadmap

**Status:** Meta-spec (Phase 1 detailed, Phase 2-5 outlined)
**Date:** 2026-04-18
**Owner:** `greentic-designer-extensions`
**Supersedes:** none
**Supersedes for planning:** none (co-exists with signing pipeline spec `2026-04-18-extension-signing-pipeline-design.md` — that spec becomes Phase 2 input)

## 1. Vision

A developer (internal Greentic engineer, enterprise customer, or community contributor) can init, develop, test, publish, and maintain a Greentic Designer Extension with near-zero friction. Benchmarks:

- Time-to-first-published-extension (fresh developer, no prior WASM experience): **≤ 30 minutes** without reading the full docs.
- Time-to-update-after-WIT-contract-bump: **≤ 5 minutes**.
- Time-to-cut-a-new-release for an existing extension: **≤ 2 minutes** (one CLI command + `git tag`).

## 2. Target Audience (hybrid)

Execution order is **internal-first** (velocity wins), then **enterprise** (signing + offline + private registries), then **community** (onboarding polish + playground + learning resources). Each subsystem is designed so the pattern fits all three; priority differs per phase.

Progress toward 10/10 DX is tracked by these metrics rather than a subjective score.

## 6. Phase 1 Architecture Overview (detailed)

Phase 1 delivers the happy path: `gtdx new → gtdx dev → gtdx publish → gtdx install → gtdx info`, end-to-end against a local filesystem registry, within 30 minutes for a fresh developer.

### 6.1 Parallel tracks + integration gate

```
┌──────────────────────────────────────────────────────────────────┐
│ greentic-designer-extensions (root crate: greentic-ext-cli)       │
│                                                                    │
│ ┌──────────────┐   ┌──────────────┐   ┌──────────────────┐       │
│ │ TRACK A       │   │ TRACK B       │   │ TRACK C           │       │
│ │ Scaffolding   │   │ Dev loop      │   │ Publish local     │       │
│ │ gtdx new      │   │ gtdx dev      │   │ gtdx publish      │       │
│ └──────┬───────┘   └──────┬───────┘   └─────────┬─────────┘       │
│        └───────────────────┼──────────────────────┘                │
│                             │                                       │
│              ┌──────────────▼───────────────┐                      │
│              │ INTEGRATION GATE              │                      │
│              │ ci/e2e/phase1_happy_path.sh   │                      │
│              └───────────────────────────────┘                      │
└──────────────────────────────────────────────────────────────────┘
```

Each track has its own sub-spec (see table in §4). The integration gate is the acceptance test.

### 6.2 Shared contract: WIT vendor pattern (frozen at Phase 1)

Problem: today developers copy WIT files with fragile relative paths (`../../wit/...`).

Decision:

- `gtdx new` copies WIT from **embedded resources** inside the `gtdx` binary (via `include_dir!`). CI in `greentic-designer-extensions` runs `cp wit/**/*.wit crates/greentic-ext-cli/embedded-wit/` before building the release binary, so each `gtdx v0.7.0` embeds the WIT contract of that version.
- Generated projects contain `wit/deps/greentic/<package>/world.wit` plus a `.gtdx-contract.lock` file recording `contract_version = "0.7.0"` + SHA256 per WIT file.
- `gtdx sync-wit` (Phase 4) reads the lock, diffs against embedded WIT in the installed `gtdx` binary, shows the diff, and offers auto-update.
- `Cargo.toml` WIT dep: `{ path = "wit/deps/greentic/extension-base", package = "greentic:extension-base@0.7.0" }` — full package spec, not file path.

### 6.3 Integration gate script

`greentic-designer-extensions/ci/e2e/phase1_happy_path.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
TMP=$(mktemp -d)
cd "$TMP"

mkdir -p local-registry
gtdx new demo --kind design --id com.example.demo
cd demo

# Build + install to local registry (skip --watch for CI)
gtdx dev --once --install-to "../local-registry"
test -f ../local-registry/com.example.demo/0.1.0/com.example.demo-0.1.0.gtxpack

# Explicit publish
gtdx publish --registry "file://../local-registry" --trust loose

# Install in fresh home
cd "$TMP"
GTDX_HOME="$TMP/gtdx-home" gtdx install --registry "file://./local-registry" com.example.demo@0.1.0 -y --trust loose
GTDX_HOME="$TMP/gtdx-home" gtdx info com.example.demo | grep "0.1.0"

echo "Phase 1 happy path: OK"
```

Target: <30s cold, <5s incremental.

## 7. Phase 1 Non-Goals (explicit)

Reviewers reject PRs that introduce these in Phase 1:

- ❌ Store HTTP publish (Phase 2, S5)
- ❌ OCI publish (Phase 2, S5)
- ❌ JCS RFC 8785 canonical signing (Phase 2, S4)
- ❌ Strict trust policy + Store countersignature (Phase 2, S4)
- ❌ Key passphrase encryption (Phase 2, S4)
- ❌ GitHub Actions template for extension repos (Phase 3, S6)
- ❌ Auto-changelog (Phase 3, S6)
- ❌ `gtdx sync-wit` (Phase 4, S7)
- ❌ `gtdx validate --against <ver>` breaking-change detector (Phase 4, S7)
- ❌ Capability deprecation model (Phase 4, S8)
- ❌ Playground / LSP / VS Code extension (Phase 5, S9)
- ❌ Hot-reload integration into consumer designer app (designer-side, out of scope for CLI repo)
- ❌ Multi-extension parallel watch (one dir = one `gtdx dev`)

## 8. Phase 1 Release Plan

**Milestone:** `gtdx v0.7.0` — "Phase 1 DX: scaffold + dev + publish (local)".

**Gates for tag:**

1. All unit + integration + e2e tests green on Linux, macOS, Windows.
2. Determinism test (`tests/deterministic_pack.rs`) stable over 10 runs.
3. `CHANGELOG.md` entry for v0.7.0 with breaking-change list (must be 0).
4. `docs/getting-started.md` (new, 5-step quickstart) merged.
5. Two reference extensions (AC design ext + `bundle-standard`) can be rescaffolded via `gtdx new` (manual verify by author).
6. Permission regression test green (local-fs install does not bypass permission prompt).
7. Coverage of new Track A/B/C code ≥ 80% (`cargo-llvm-cov`).

**Rollout:**

1. Merge to `main` after three tracks + integration gate pass review.
2. Tag `v0.7.0`, CI builds binary for linux/macos/windows.
3. Publish GitHub Release (existing workflow).
4. Announce in internal channel.
5. Update memory `designer-extensions-system.md` with "Phase 1 shipped" note.

## 9. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `cargo component` breaking change mid-phase | Medium | High | Pin `cargo-component = 0.21.1` in preflight; document upgrade path |
| Deterministic ZIP fails on Windows (CR/LF) | Medium | Medium | Normalize line endings in packer; test on Windows CI |
| Platform variance Windows file watcher | Medium | Medium | Docs + default debounce 1000ms on Windows |
| Signing partial at Phase 1 misinterpreted as production | High | High | `--sign` emits loud warning "Phase 1 signing has known verification gap" |
| Scope creep into Phase 2+ | High | Medium | Non-goals list (§7) is enforced in reviews |
| WIT embed size inflates binary | Low | Low | 5 files × ~3 KB = ~15 KB; negligible |
| User confusion local-fs vs Store | Medium | Medium | Dedicated `docs/registries.md`; clear `gtdx registries list` output |

## 10. Phase 1 Exit Criteria (Definition of Done)

Phase 1 is **done** when all of the following hold:

1. `gtdx new demo --kind design` works on Linux/macOS/Windows; generated project passes `cargo check`.
2. `gtdx dev --watch` detects an edit, rebuilds, and reinstalls in ≤ 5 s (debug, incremental).
3. `gtdx publish --registry local` produces a byte-identical `.gtxpack` on repeat runs (determinism).
4. `gtdx install` from local registry → `gtdx info` shows correct metadata.
5. Both reference extensions (AC design ext, `bundle-standard`) rescaffolded via `gtdx new` compile without manual fixes.
6. E2E script green on all CI OS matrix entries.
7. Coverage of new code ≥ 80%.
8. `docs/getting-started.md` merged.
9. WASM unit harness (`greentic-extension-sdk-testing::wasm_harness`) published with ≥ 1 integration test in AC design ext.
10. Memory updated (`designer-extensions-system.md`).

**Outcome metric:** A volunteer internal developer runs a stopwatch from `gtdx new` to `gtdx info` showing their scaffolded extension — target ≤ 15 minutes (stretch: ≤ 10 minutes).

## 11. Downstream Phases (Outlines)

### Phase 2 — S4 + S5 (signing + remote publish)

- Integrate JCS RFC 8785 canonical signing (existing `2026-04-18-extension-signing-pipeline-design.md` spec becomes Phase 2 input).
- Wire `gtdx keygen`/`sign`/`verify` subcommands (already in `feat-signing` worktree).
- Implement `GreenticStoreRegistry::publish()` (HTTP) against Store server OpenAPI.
- Implement `OciRegistry::publish()` via `oras` or native OCI client.
- `gtdx login` / `gtdx logout` for Store auth; per-registry credentials in `~/.greentic/credentials.toml`.

### Phase 3 — S6 (release automation kit)

- `greentic-designer-extensions/templates/github-actions/ci.yml` + `release.yml` injected by `gtdx new`.
- Auto-changelog generation from Conventional Commits (via `git-cliff` or similar).
- `gtdx release` command that drives: version bump, tag, push, trigger CI → CI calls `gtdx publish`.

### Phase 4 — S7 + S8 (WIT sync + deprecation)

- `gtdx sync-wit`: diff project's `.gtdx-contract.lock` vs installed `gtdx` embedded WIT. Interactive update.
- `gtdx validate --against <version>`: fetch prior version from registry, check WASM ABI + capability version compatibility.
- Capability deprecation model: `deprecated: true` in describe.json with `superseded_by` field; runtime emits warning when consumers resolve deprecated capabilities.

### Phase 5 — S9 (polish tier)

- `gtdx doctor` real checks (toolchain versions, registry reachability, WIT contract currency, permissions dir).
- `gtdx playground` (optional): spawn a local designer stub that auto-subscribes to local registry so dev loop is one-command.
- Error message upgrades across all subcommands (JSON pointers, suggestions).
- VS Code extension: `describe.json` schema auto-complete + validation.
- Opt-in anonymous telemetry for usage metrics.

## 12. Cross-Cutting Principles

- **JSON output option** (`--format json`) on every CLI for IDE / CI integration from Phase 1 onward.
- **No destructive defaults**: `gtdx install`, `gtdx publish`, `gtdx uninstall` never overwrite without confirmation or `--force`.
- **Atomic writes**: all registry mutations via tempfile + `rename(2)`.
- **Reproducible outputs**: deterministic ZIP packing, sorted entries, zeroed timestamps.
- **Bahasa Indonesia hanya untuk interactive end-user CLI output** (following user preference). Log messages, errors, docs, code comments stay in English.
